本文件依你確認的決策整理。實作：`npm.cmd run import:xlsx` → `data/import-from-xlsx.json` + `data/import-validation-report.json`。

來源檔：`data/RMB帳本.xlsx`

## 已確認決策

| # | 項目 | 決策 |
|---|------|------|
| 1 | 資料來源 | **資產流向表**為寫入依據；**各客戶草表**（柏草、胡草…）僅做**交叉驗證**，不雙寫 |
| 2 | 時間範圍 | **全部**歷史 |

| 7 | 買入／廠商 | 廠商名稱自資料**同步**；試算表內買入皆視為 **`paymentStatus: paid`**（已付台幣） |
| 8 | 精度目標 | **可操作近似**：允許少數筆差異，以驗證報告＋備註標示，不強求分毫不差 |

## 試算表結構摘要

### 寫入依據：`資產流向表`

交易區欄位（自「日期｜類別｜對象/客戶…」表頭列起）：

| 欄 | 試算表 | 用途 |
|----|--------|------|
| A | 日期 | `createdAt` |
| B | 類別 | 決定匯入動作 |
| C | 對象/客戶 | 客戶名（售出/收帳）或廠商名（買入） |
| D | 草 | RMB 數量 |
| E | 匯率 | 售出/買入匯率 |
| F | 應收 | 台幣應收（售出） |
| G | 收帳 | 台幣收帳金額（若該列為收帳類） |
| H | 台 | 台幣金額（買入付款等） |
| I | 出帳戶 | 扣款帳戶 |
| J | 入帳戶 | 入帳帳戶 |
| K | 備註 | `description` / 差異說明 |

**類別統計（約）**：售出 222、收帳 118、買入 23、內轉 25、增資 1。頂部「持有台／持有草」為彙總，**不當交易匯入**。

**買入備註**：多數列 `對象/客戶` 為空，僅少數為「許」；台幣多由 `草 × 匯率` 推算，`出帳戶`→`入帳戶` 表示付台幣、入 RMB。廠商為空時，匯入規則見下文。

**帳戶字串（去重）**：`0107台`、`0107草`、`6186台`、`6186草`、`7773台`、`7773草`。

### 驗證用：客戶草表（柏草、胡草、芸草、吳草、蕭草）

欄位：`日期 | 草 | 匯率 | 成本 | 應收 | 利潤 | 收帳 | 欠款 | 備註`

- 有 **草＋匯率** → 視為售出列（與流向表「售出」對帳）
- 有 **收帳** → 視為收帳列
- 僅 **欠款**、無草/收帳 → 期初或結轉列（用於對最後欠款）

### 不匯入

- **俐草**（欄位不同：儲值草／花費／欠款）
- **分帳用**（拆帳、散客利潤，系統無對應模組）

## 系統對照（RMBsale）

匯入目標為 `BusinessDataImport`（見 `src/lib/dataImport.ts`），經管理後台 JSON 匯入或同結構檔案：

`holders`, `accounts`, `customers`, `channels`, `purchases`, `sales`, `saleAllocations`, `rmbLots`, `ledger`

建議以 **重播 `localStore` 業務函式** 產生資料（與日常操作一致），而非手填不一致的流水。

## Phase 0：解析與主檔

### 0.1 帳戶（同步命名）

- 自流向表 **出帳戶 / 入帳戶** 及頂部持有列，收集所有帳戶字串（排除表頭文字「出帳戶」「入帳戶」）。
- 每個字串 → 一筆 `accounts.name`；`currency` 依尾字：**草 → RMB**，**台 → TWD**。
- **持有人**：以名稱前綴數字為 `holders.name`（如 `0107`、`7773`、`6186`），同一前綴共用一個 holder；若無法解析則 holder 名稱與帳戶名相同。
- 期初餘額：優先讀「持有台／持有草」彙總列；不足時在 Phase 3 以差異報告標註，必要時以調整流水補平（需你事後確認）。

### 0.2 客戶

- 自流向表「售出」「收帳」的 **對象/客戶** 去重 → `customers.name`（保留 **柏草** 等完整字串）。
- 草表客戶名應與此集合一致；草表獨有而流向表無者 → 驗證報告 **warning**（不寫入）。

### 0.3 廠商（channels）

- 自流向表 **類別=買入** 的 **對象/客戶** 去重；空字串匯入為單一 channel：**`未指定`**（或備註內廠商名，若有）。
- 表中出現的 **許** 等名稱原樣建立 `channels.name`。
- 所有買入：`paymentStatus: "paid"`，`paidTwd` = `twdCost`（台幣欄位若空則 `草 × 匯率`，四捨五入與試算表一致）。

## Phase 1：依流向表重播交易（寫入唯一來源）

依 **日期＋列號** 排序，略過彙總列與重複表頭。

| 類別 | 系統行為 |
|------|----------|
| **買入** | `purchase` + `rmbLot` + RMB 入帳（入帳戶）+ TWD 出帳（出帳戶）+ 應付／應付付款（已付清） |
| **售出** | `sale` + FIFO 分配 + RMB 出帳 + 應收 + 利潤流水 |
| **收帳** | 客戶應收減少 + TWD 入帳（依出/入帳戶欄位） |
| **內轉** | 帳戶轉帳（對應既有內轉邏輯） |
| **增資** | TWD 入金（或對應增資流水類型） |

無法對應帳戶名、FIFO 庫存不足、客戶不存在等 → 該列 **跳過並寫入驗證報告**（不中止整批），符合「可操作近似」。

**手續順序**：買入 → 售出 → 收帳 → 內轉 → 增資（同類內依日期）。

## Phase 2：交叉驗證（不雙寫）

對每位客戶草表（柏草、胡草、芸草、吳草、蕭草）：

| 檢查項 | 作法 | 通過標準（近似） |
|--------|------|------------------|
| 售出筆數 | 草表「有草」列數 vs 流向表該客戶「售出」列數 | 相等或差異 ≤ 約定筆數（預設 3），否則 note |
| 售出金額 | 草 × 匯率 或應收欄合計 vs 流向表應收合計 | 差額 &lt; 1 TWD 或相對誤差 &lt; 0.01% → OK；否則 note |
| 收帳 | 草表收帳合計 vs 流向表收帳合計 | 同上 |
| 期末欠款 | 草表最後一列 **欠款** vs 系統 `customer.receivableTwd` | 差額 &lt; 1 TWD → OK；否則 note |
| 客戶存在 | 草表標題名是否在 `customers` | 必須存在 |

彙總輸出：`import-validation-report.json`（或 Markdown），含 **OK / WARN / SKIP** 與建議備註欄位。

## Phase 3：交付與匯入

1. 執行轉換腳本 → `data/import-from-xlsx.json` + 驗證報告  
2. 你審閱 WARN 列表，必要時在試算表或備註補說明  
3. 管理後台：**清除數據** → **匯入 JSON**（或腳本直接呼叫 `replaceBusinessData` 同等效果）  
4. 在 App 抽查：儀表板總額、各客戶應收、RMB 帳戶、流水筆數

## 實作順序（開發用）

1. `scripts/xlsx-to-import.mjs`：解析 + Phase 0/1 + 輸出 JSON  
2. `scripts/validate-xlsx-import.mjs`：Phase 2 報告（可併入上一支）  
3. 單元測試：固定 2～3 筆買入/售出/收帳樣本，對 golden JSON  
4. （選）管理後台「從試算表產生 JSON」按鈕 — 延後  

開發用探測腳本：`scripts/probe-xlsx.mjs`（已用於結構分析）。

## 風險與已知限制

- **買入廠商欄位常空**：以 `未指定` channel 匯入，報告中列出；你日後可在後台改名合併。  
- **FIFO**：歷史售出若早於買入日期，可能觸發庫存不足 → 該筆 SKIP + note。  
- **成本／利潤**：草表「成本」「利潤」欄多為空，**不以草表覆寫**系統計算值。  
- **俐草／分帳**：完全不進系統；相關金額不計入驗證總計。  
- **柏草 3000 列**：僅參與驗證，不逐列寫入。
